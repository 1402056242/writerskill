# 正文 (Write)

## 目标
将大纲转化为高质量正文，按章纲分场景撰写，并更新角色状态、伏笔与进度，同时为每一章生成一份**章节提交文件**（chapter commit），作为后续一切视图和分析的事实源头。

> **🚨 产出限制（白名单）**：
> - 在 `4-正文/` 下**仅**允许创建或覆盖：
>   - `第X章_草稿.md`
>   - `第X章_正文.md`
>   - `prompts/第X章_写作提示词.md`
>   - `prompts/第X章_校准提示词.md`
> - 在 `.novelkit/story/commits/` 下**仅**允许创建或覆盖：
>   - `chapter_{X}.commit.json`
> - **禁止**擅自新增其他正文类 Markdown 或任意命名的 JSON 文件。

## 📌 阶段开始前：回顾预期

**【立即提示用户】**：
> 进入正文阶段前，建议先打开 `1-边界/预期.md`，回顾你的初始执念与边界，确认正文写作方向与预期一致，再开始动笔。

---

## 步骤 0: 一次性初始化 (One-time Setup)

- **目标**：以主角的初始状态作为整个创作阶段的基准。（插件初始化时已创建相关文件，本步仅做更新与链接。）
- **动作**：首次创作前，根据 `XX卷_章纲.md` 的开局设定：
  1. **更新初始状态**：在 `.novelkit/memory/character_state.md` 中记录/更新主角第 0 章（正文开始前）的初始状态。
  2. **链接到 SOLOENT**：确认 `SOLOENT.md §7.2 Character Snapshot` 已写入指针 `-> .novelkit/memory/character_state.md`；若未写入则补上。

---

## 步骤 1：生成写作提示词 (Build Prompt)

- **目标**：通过脚本自动组装本章所需的所有上下文（包括章纲、前三章正文、角色状态、文风与宪法等），防止 AI 漏读或上下文过载。
- **动作**：AI 执行以下命令生成提示词文件（将 `<章号>` 替换为当前要写的章节数字，如 `15`）：
  ```bash
  python3 .soloent/skills/wuhang-long-all/scripts/build_writing_prompt.py <章号>
  ```
- **结果**：脚本会在 `4-正文/prompts/` 目录下生成 `第X章_写作提示词.md`。
- **【强制暂停】** 提示词生成后，**禁止自动进入步骤 2，等待用户明确回复**：
  > 「第 X 章的写作提示词已生成（`4-正文/prompts/第X章_写作提示词.md`）。你可以打开文件检查或手动修改提示词，确认无误后回复『继续』，我将开始撰写正文。」
- 收到用户『继续』（或等效肯定）后，方可进入步骤 2。

---

## 步骤 2：执行正文撰写 (Writing with AI)

- **输入**：读取刚刚生成的 `4-正文/prompts/第X章_写作提示词.md`。
- **动作**：
  - **读取并执行提示词**：AI 将 `4-正文/prompts/第X章_写作提示词.md` 的全文内容作为本次创作的**唯一核心指令**。
  - **严格遵循提示词**：必须严格按照提示词文件中的【六、写作输出要求】进行创作（分场景、字数、文风、结尾钩子等）。
  - **直接输出**：不需要任何开场白、思考过程或解释，直接输出正文内容。
- **输出产物**：将生成的正文写入 `4-正文/第X章_草稿.md`。
- **【强制暂停】** 草稿生成后，**禁止自动进入步骤 3，等待用户明确回复**：
  > 「第 X 章草稿已生成（`4-正文/第X章_草稿.md`），请阅读确认后回复『继续』进行完稿校准。」
- 收到用户『继续』（或等效肯定）后，方可进入步骤 3。

---

## 步骤 3：完稿后校准 (Post-Writing Calibration)

- **目标**：通过脚本自动统计字数并组装校准提示词，让 AI 扮演严苛主编对草稿进行五项审查，输出可用于后续提交的健康标记。
- **动作**：AI 执行以下命令生成校准提示词文件（将 `<章号>` 替换为当前章节数字）：
  ```bash
  python3 .soloent/skills/wuhang-long-all/scripts/build_calibration_prompt.py <章号>
  ```
- **结果**：
  - 脚本会在 `4-正文/prompts/` 目录下生成 `第X章_校准提示词.md`；
  - 并在终端输出当前草稿的中文字数。
- **执行校准**：
  - AI 读取 `4-正文/prompts/第X章_校准提示词.md`，严格按照其中的五项审查标准，显式输出一份**【完稿自检卡】**。
  - 自检卡中应包含：节奏、爽点密度、逻辑一致性、角色不崩、人称/视角等维度的结论与修改建议，可为后续章节提交文件中的 `health_flags` 提供数据来源。
- **分支处理**：
  - **若审查未通过**：AI 必须根据自检卡中的修改建议，修订草稿，并再次执行步骤 3，直到五项全部通过（或用户明确接受当前版本的缺陷）。
  - **若审查通过**：AI 将 `第X章_草稿.md` 重命名为 `第X章_正文.md`，并提示用户：
    > 「第 X 章完稿自检已通过，正文已落定。请确认无误后回复『继续』，我将进行记忆与状态回写，并生成章节提交文件。」
- 收到用户『继续』（或等效肯定）后，方可进入步骤 4。

---

## 步骤 4：伏笔、状态与进度更新 (Foreshadowing, State & Progress Update)

⚠️ **CRITICAL STEP**：此步骤决定了记忆的连续性和后续章节提交文件的基础数据，绝对不可跳过！

> **🚨 增量更新铁律 (Strict Append & Update Rule)**：
> 每次更新 memory 文件时，**只允许添加新信息或修订发生变化的信息**。
> **绝对禁止删除或遗漏**任何与本章无关的既有信息（如之前积累的人物状态、未收回的伏笔等）。必须在保留原有全部内容的基础上进行增量修改！

- **状态回写**：根据本章发生的实际剧情，**立即更新**以下文件：
  - `.novelkit/memory/character_state.md` → 记录/更新角色状态变化（升级/受伤/获得新道具/结成或破裂的关系等）。**（遵守增量铁律：原有角色和属性不得丢失）**
  - `.novelkit/memory/foreshadowing.md` → 新增或收回的伏笔条目，标记每条伏笔的状态（未收回/已收回）、相关章节号和简要说明。**（遵守增量铁律：未涉及的伏笔必须原样保留）**
  - `SOLOENT.md §8` → 将本章标记为已完成，并填入下一章的简要任务或风险提示。
- **与后续章节提交的关系**：
  - 本步骤中整理出的**角色状态变更**与**伏笔变更**，将直接用于构建章节提交文件中的 `character_state_diff` 与 `foreshadowing_diff`。
  - 若本步骤未准确整理这些信息，后续 chapter commit 将失去可信度。

更新完成后，再进入章节提交步骤。

---

## 步骤 4.5：章节提交与投影 (Chapter Commit & Projection)

- **目标**：为本章生成一份不可变的**章节提交文件**，作为后续一切视图（memory、dashboard、统计分析等）的唯一事实来源。
- **产物目录**：`.novelkit/story/commits/`
- **文件命名**：`chapter_{X}.commit.json`（例如：`chapter_15.commit.json`）

### 提交内容要求（字段约定）

该 JSON 文件至少应包含以下字段（字段名可根据脚本实现微调，但语义必须保持）：

- `chapter_id`：本章唯一标识（通常为整数章节号，如 `15`）。
- `title`：章节标题（与 `第X章_正文.md` 中标题一致）。
- `word_count`：本章最终正文字数（可由校准脚本统计）。
- `events`：本章发生的关键事实列表。建议按结构化格式记录，例如：
  - 事件 ID / 相关角色 / 时间地点 / 行为 / 结果 / 后续影响。
- `character_state_diff`：本章对主要角色状态的**增量变更**：
  - 每条记录应包含角色标识、变更前状态概要、变更后状态概要，或至少「关键属性变化摘要」。
- `foreshadowing_diff`：本章对伏笔系统的**增量变更**：
  - 新增伏笔（含简要描述、预计收回章节或阶段）。
  - 推进伏笔（补充信息或提升紧张度）。
  - 收回伏笔（指明对应的埋入来源章节）。
- `health_flags`：从完稿自检卡中提取的健康标记：
  - 例如：`["pacing_ok", "hook_strong", "logic_clean"]` 或 `["risk_slow_start"]`。
- `created_at`：章节提交时间戳（建议使用 ISO8601 格式）。

> **说明**：`events` / `character_state_diff` / `foreshadowing_diff` 的具体结构由脚本负责写入，但 AI 必须在对话与 memory 更新中提供足够精确的信息，以支持这些字段的构建。

### 投影规则与一致性要求

- **唯一事实源头**：
  - 自本步骤起，`.novelkit/story/commits/chapter_{X}.commit.json` 视为本章事实的唯一权威来源。
  - 后续如需构建 Dashboard、追读力系统、统计图表等，只能从 commits 中读取数据，而不能直接扫描正文 Markdown。
- **memory 与 commit 的关系**：
  - `.novelkit/memory/character_state.md` 与 `.novelkit/memory/foreshadowing.md` 可视为 commits 的「聚合视图」：
    - memory 文件中记录的是**当前最新状态**；
    - 每个 chapter commit 记录的是**本章的状态差分**。
  - 更新 memory 时必须确保：
    - 不与最近一条 commit 中的 `character_state_diff`、`foreshadowing_diff` 冲突；
    - 未在 commit 中出现的旧信息不得被删除或隐性修改。
- **回放与修复的可能性（预留）**：
  - 未来如需实现「重放投影」脚本，可通过顺序读取 `chapter_1.commit.json` → `chapter_2.commit.json` → ... → `chapter_N.commit.json` 来重建任意时刻的 memory 状态。
  - 因此，commit 的字段和增量信息必须足够完整，避免依赖运行时的隐含上下文。

章节提交文件生成并写入成功后，方可视为本章在「Story System」主链上的一次正式入账。

## ⏭️ 收口与下一步 (Loop Control)

- **动作**：在完成步骤 1–4.5 后，**必须暂停**，等待用户明确指令，严禁自动开始下一章。
- **状态检查与分支选择**：
  1. **正常推进**：
     - 检查当前卷 `XX卷_章纲.md`。
     - **有剩余章节**：询问「第 X 章完成并已提交。是否继续第 X+1 章？」  
       - 若用户同意，则返回步骤 1，为下一章生成写作提示词。
     - **本卷已完**：提示「本卷完。请加载 **wuhang-outline**（`wuhang-outline.md`）规划下一卷，或先对本卷进行复盘与修订。」
  2. **剧情偏离 / 修订需求**：
     - 若本章剧情与章纲差异较大，或用户表示「有了新想法」：
       - **引导**：「检测到/你提到剧情有变化，是否加载 **wuhang-outline**（`wuhang-outline.md`）的【动态修订】模块，重做后续大纲？」  
       - 如用户同意，暂停写作 loop，转入大纲修订流程。
- **正文质检建议**：
  - 每章初稿完成后，可加载 **wuhang-review**（`wuhang-review.md`，模块 3：正文质检）检查黄金三章与代入感；质检通过后再继续下一章循环。
  > **说明**：目前的 review 流程由武行与 AI 交互生成，仅作为简单参考，暂不具备更实际和更有经验的指导意义，后续有时间会更新真正意义的 review。
