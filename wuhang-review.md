# 审查 (Review)

## 目标
在设定、大纲、正文等关键节点进行质量把控，防止问题累积；支持设定回溯（剧情与设定两张皮时更新设定），并在需要时将审查结果同步到章节提交文件（chapter commits）中，形成可追踪的健康标记。

> **📌 与 Story System 的关系**：
> - 设定/大纲/正文审查并不会直接修改 `.novelkit/story/commits/chapter_{X}.commit.json` 中的历史事实，但可以更新其中的 `health_flags` 等元信息；
> - 这样，后续 Dashboard 或统计脚本可以按「通过审查 vs 有风险」对章节进行筛选和展示。

---

## 📌 阶段开始前：回顾预期

**【立即提示用户】**：
> 进入审查阶段前，建议先打开 `1-边界/预期.md`，以初始预期为基准来审视当前产出，检查方向是否跑偏。

---

> **⚠️ 说明**：本质检流程由 AI 生成，目前正根据武行个人创作经验进行深度优化。以下标准仅作为参考建议，请根据实际情况灵活调整。  
> **原则**：早发现，早治疗。设定服务于剧情，剧情服务于爽点，所有变更最终应在章节提交和 memory 中有所体现。

## 🛠️ 模块 1: 设定质检 (Setting Review)

**触发时机**：完成创意与设定流程，生成设定文件后。  
**检查对象**：

- `2-设定/2.1_创意脑暴.md`
- `2-设定/2.2_新书设定案.md`
- `2-设定/2.3_金手指设定.md`
- `2-设定/2.4_主要角色设定表.md`

**质检标准 (Checklist)**：

1. **逻辑自洽性 (Internal Logic)**
   - [ ] **力量体系闭环**：金手指的代价与收益是否平衡？后期是否会战力崩坏？
   - [ ] **世界观支撑**：地图广度与等级上限是否匹配？换地图逻辑是否顺畅？
   - [ ] **稀缺性驱动**：世界观的核心资源是否有稀缺性？谁垄断了它？底层为打破垄断付出了什么代价？（无稀缺 = 无冲突 = 无故事）
   - [ ] **链式反应自查**：核心设定（魔法/金手指/怪物）是否对生产力、文化、政治产生了连带影响？还是只贴了个标签、世界照旧运转？

2. **爽点支撑度 (Pleasure Points)**
   - [ ] **金手指独特性**：是否只有主角能用？是否具备「人无我有，人有我优」的绝对优势？
   - [ ] **期待感设计**：金手指的进化路线是否清晰？是否有足够的升级期待？
   - [ ] **噱头商业自查**：选题噱头能用一句话说清楚吗？是市场有共识的题材，还是只有小圈子懂的？书名/简介与实际内容的期待感是否一致？

3. **利益驱动逻辑 (Motivation)**
   - [ ] **主角驱动力**：主角的目标是否足够强烈（生存/复仇/长生）？
   - [ ] **反派智商**：反派的行为是否符合其利益最大化？拒绝无脑送经验。反派在已知信息下的决策，是否是合理的最优解？
   - [ ] **文明共识对齐**：NPC 的反应与价值观是否符合那个世界的集体潜意识？（修仙世界路人不应有现代人权意识）

4. **人物立体度 (Character Depth)**
   - [ ] **反差萌/记忆点**：主角和核心配角是否有独特的标签或反差（A 面/B 面）？
   - [ ] **记忆钢印**：每个重要配角是否有一个极简的视觉或性格标签（如：永远在咳嗽的军师、剥花生的胖子）？读者记不住名字，但要记住标签。
   - [ ] **群像冲突链**：如果有多人场景，这些人的性格和手段是否有摩擦？是否避免了「你说一句、我赞同、他补充」的复读机对话？

**执行动作**：

- 若发现问题，直接在对话中指出，并协助用户修改对应设定文件。
- 修改后，须再次确认是否引入新矛盾。
- 如本次设定质检会影响已写章节，应提醒用户：后续需要在写作/修订时同步更新章节提交和 memory。
- **【强制暂停】** 质检完成后，**禁止继续执行任何后续操作，等待用户明确回复**：
  > 「设定质检已完成，请查看以上问题与建议。确认修改完毕或选择忽略后回复『继续』。」
- 收到用户『继续』（或等效肯定）后，可进入大纲质检或直接回到大纲/写作阶段。

## 📋 模块 2: 大纲质检 (Outline Review)

**触发时机**：完成大纲规划流程，生成总纲、卷纲或章纲后。  
**检查对象**：

- `3-大纲/3.1_全书结构总纲.md`
- `3-大纲/XX卷_完整卷纲.md`
- `3-大纲/XX卷_章纲.md`

**质检标准 (Checklist)**：

1. **节奏密度 (Pacing)**
   - [ ] **黄金节奏**：是否符合「三章一小高潮，十章一大高潮」的网文节奏？
   - [ ] **垃圾时间**：是否存在连续 2 章以上的铺垫或说明性文字？（须压缩或删除）
   - [ ] **快慢倒置检查**：铺垫是否占用了太多篇幅？读者最想看的那一秒（高潮爆发点）是否被慢放、用五感拉长？还是反而三句话带过？
   - [ ] **高潮后喘气口**：每个大高潮后是否安排了闲笔（景物/小日常），给读者情绪落地的空间？（一直绷紧的弦会断）

2. **期待感管理 (Hooks)**
   - [ ] **钩子有效性**：每一章（特别是卷末/章末）是否留下了强悬念？
   - [ ] **伏笔埋设**：高潮前的铺垫是否足够？是否有生硬的机械降神？
   - [ ] **信息差运用**：是否有「读者知道的比主角多（紧张感）」或「主角知道的比读者多（悬念感）」的设计？两者交替出现会让读者欲罢不能。
   - [ ] **噱头兑现度**：书名/简介承诺的核心体验（复仇/无敌/爽/恋爱），在本卷大纲中是否有具体事件承接？还是被「晾」在那里？

3. **爽点逻辑 (Payoff)**
   - [ ] **情绪释放**：压抑后的爆发是否足够酣畅？
   - [ ] **主角作用**：核心转折点是否由主角主导？（拒绝主角划水）
   - [ ] **三段式给料**：重要信息是否分层释放（预警→加重→一刀扎心），而不是一鼓脑说完？

4. **一致性检查 (Consistency)**
   - [ ] **设定对齐**：大纲剧情是否与 `2.2_新书设定案.md` 中的主线推演一致？
   - [ ] **战力检查**：本卷 BOSS 战力是否在主角当前等级的可应对范围内？

5. **群像结构 (Ensemble)**
   - [ ] **共同灵魂**：如有群像，这群人是否有共同底色（共同目标/信念/宿命），使他们聚在一起有说服力？
   - [ ] **宿命感设计**：群像成员的结局是否各不相同？终点的对比是否会产生厚重的时代感？

**执行动作**：

- 模拟【主编】视角，对大纲进行辣评。
- 提出具体的修改方案（如：调整事件顺序、增加危机感、强化金手指作用）。
- 如大纲质检发现问题，应提示用户考虑使用 `wuhang-outline` 的「动态修订」模块，并在必要时同步更新相关设定文件。
- **【强制暂停】** 质检完成后，**禁止继续执行任何后续操作，等待用户明确回复**：
  > 「大纲质检已完成，请查看以上问题与建议。确认修改完毕或选择忽略后回复『继续』。」
- 收到用户『继续』（或等效肯定）后，可进入正文质检或回到大纲修订/写作阶段。

## ✍️ 模块 3: 正文质检 (Draft Review)

**触发时机**：完成正文写作流程，生成单章正文草稿或正稿后。  
**检查对象**：

- `4-正文/第X章_草稿.md` 或 `4-正文/第X章_正文.md`
- （可选）`.novelkit/story/commits/chapter_{X}.commit.json`（用于对照事件与健康标记）

**质检标准 (Checklist)**：

1. **黄金三章法则 (Opening)**（仅限开篇）
   - [ ] **切入点**：是否开篇即危机/冲突？（拒绝慢热）
   - [ ] **金手指**：金手指是否在合适时机（通常前 3 章）出现并发挥作用？
   - [ ] **代入感**：主角的处境是否能引起读者共情？
   - [ ] **噱头对板**：开篇实际呈现的内容，是否与书名/简介给读者建立的期待一致？

2. **节奏控制 (Pacing)**
   - [ ] **快慢分配**：铺垫是否控制在 3 句以内带过？读者最期待的那一刻（信息揭露/爽点爆发）是否用五感慢放拉长？
   - [ ] **视觉节奏**：打斗/追逐/生死一线时，是否把长句拆碎、多换行，让阅读速度跟着心跳加速？慢情绪段落是否改用长句大段？
   - [ ] **喘气口**：大高潮后是否有 1–2 句闲笔（景物/小动作）让读者情绪落地，再拉紧下一个悬念？
   - [ ] **信息差**：本章是否有「读者知道主角不知道」或「主角知道读者不知道」的时刻？

3. **对话质量 (Dialogue)**
   - [ ] **身份化语言**：不同身份/性格的角色，说话逻辑是否截然不同？市井小民、读书人、杀手绝不能说同一种话。
   - [ ] **潜台词与张力**：对话中是否有「词不达意/指桑骂槐/说反话」？角色是否在直白表达意图？（想要钱的要谈梦想，恨入骨的说话要温润如玉）
   - [ ] **打破有问必答**：是否存在大量一问一答的乒乓球式对话？是否运用了答非所问、反问压制、情绪切断中的至少一招？
   - [ ] **动作辅助**：对话间是否穿插了人物细微动作（撒谎时的小动作、紧张时的失误）？动作的画面对比是否比描写脸色更高级？

4. **动作戏质量 (Action)**（有打斗场景时）
   - [ ] **动词颗粒度**：是否把模糊的「打/踢/跑」拆解成有重量/方向感的具体动词（砸/轰/撕/钻/绕）？
   - [ ] **侧面烘托**：力量感是否通过受击者和环境的反馈来体现，而不是直接说「这一拳有万斤之力」？
   - [ ] **五感临场**：是否调动了听觉（刀锋划骨声）、嗅觉（血腥/火药）、触觉（虎口震裂/汗水刺眼）？读者是旁观者还是钻进了主角身体？
   - [ ] **博弈感**：每一招出击是否有明确目的（察觉对方旧疾/卖破绽设陷阱/利用环境）？是智商在线的打斗，而不是纯粹对轰？

5. **沉浸感 (Immersion)**
   - [ ] **Show, Don't Tell**：是否用动作、对话、细节描写代替了枯燥的设定说明？
   - [ ] **五感描写**：是否调动了视觉、听觉、嗅觉等感官体验？

6. **情绪调动 (Emotion)**
   - [ ] **爽点/憋屈点**：读完本章，读者是感到爽、期待，还是憋屈？（憋屈不能过夜）
   - [ ] **装逼打脸**：如果有装逼情节，是否铺垫到位？打脸是否响亮？

7. **文风与红线 (Style & Constraints)**
   - [ ] **AI 味去除**：检查是否存在翻译腔、重复词汇、过度书面语，以及 `MASTER.md` 负向词库中的禁用词汇。
   - [ ] **红线检查**：是否违反项目宪法（`.novelkit/constitution/MASTER.md`）中的禁忌？

8. **与章纲/设定的一致性**
   - [ ] 是否遵守当前卷 `XX卷_章纲.md` 中的事件顺序与伏笔安排？
   - [ ] 是否与 `2.2/2.3/2.4` 中的世界观、金手指、人设吻合？

**执行动作**：

- 模拟【毒舌读者】或【资深编辑】视角，进行段落级批注。
- 重点标注「出戏」、「啰嗦」、「逻辑不通」的段落，并提供参考改写方案。
- 如存在严重问题，可建议回到 `wuhang-outline` 或 `wuhang-ideation` 做结构级/设定级修订。

### 与章节提交文件的联动

如果本次审查针对的章节已经有对应的 `chapter_{X}.commit.json`（或即将生成）：

- 可以基于审查结果给出本章的健康标记，例如：
  - `["pacing_ok", "hook_strong", "logic_clean", "dialogue_layered", "action_visceral"]`
  - 或带风险提示：`["risk_slow_start", "risk_thin_hook", "risk_flat_dialogue", "risk_direct_force_description"]`
- 建议在审查结束后由脚本或 AI 帮助更新该 commit 中的 `health_flags` 字段：
  - 若 commit 尚未生成：可在 `wuhang-write` 的章节提交步骤中，将本次审查意见转写为 `health_flags`；
  - 若 commit 已存在：通过工具或脚本合并新的 `health_flags`，保证历史事实不被修改，仅补充元信息。

> 原则：审查不会改写历史事件，只会为章节提交文件添加或更新健康标记，使后续 Dashboard/统计能识别「已审+健康」与「待审/有风险」章节。

### 用户确认

- **【强制暂停】** 质检完成后，**禁止继续执行任何后续操作，等待用户明确回复**：
  > 「正文质检已完成，请查看以上批注与建议。确认修改完毕或选择忽略后回复『继续』。如本章已有章节提交文件，请在后续步骤中同步更新 `health_flags`。」
- 收到用户『继续』（或等效肯定）后，方可继续下一章 `wuhang-write` 循环，或进入大纲/设定回溯。

## 🔄 模块 4: 设定回溯 (Feedback Loop)

**定位**：防止「剧情–设定两张皮」，确保设定随剧情动态演进，并逐步体现在后续章节提交和 memory 中。

**触发时机**：

- 正文/大纲写作中涌现新灵感，与旧设定冲突时。
- 发现旧设定无法支撑当前剧情爽点时。
- 审查模块发现「设定吃不下当前爽点」或「角色行为与既定人设矛盾」。

**执行动作**：

1. **更新 Memory Bank**：
   - 补充新配角 -> `SOLOENT.md §3` & `§7.2` 与 `.novelkit/memory/character_state.md`
   - 修正地图 -> `SOLOENT.md §2` 与 `2.2_新书设定案.md`
   - 更新伏笔 -> `SOLOENT.md §7.4` 与 `.novelkit/memory/foreshadowing.md`

2. **更新设定文档**：
   - `2-设定/2.2_新书设定案.md`（主线/地图/关键事件）
   - `2-设定/2.3_金手指设定.md`（补丁/进化）
   - `2-设定/2.4_主要角色设定表.md`（性格成长/关系变化）

3. **对章节提交的影响提示**：
   - 如果设定变更影响到已写章节的理解（例如：某个事件的动机或世界观解释），应提示用户：
     - 可以在后续章节提交时增加注解事件，或在新的 commit 中通过事件记录「设定补丁」；
     - 暂不推荐直接回写旧 commits 的事实字段，以免破坏历史记录；如需改旧章，应通过重写章节 + 新 commit 的方式进行。

## ✅ 质量检查表 (Total Quality Control)

- [ ] **设定层**：逻辑闭环，爽点明确，人设鲜活，世界观有稀缺性驱动。
- [ ] **大纲层**：节奏紧凑，期待感强，结构稳固，信息差设计到位。
- [ ] **正文层**：代入感强，情绪到位，对话有张力，动作有画面感，文风达标。
- [ ] **数据层**：字数达标，Memory Bank 已更新，该阶段涉及的章节提交文件已补充必要的 `health_flags`。

## 📋 审查阶段报告（Review Stage Report）

每次审查模块结束（设定 / 大纲 / 正文），建议输出简单报告：

- **模块**：设定 / 大纲 / 正文
- **状态**：通过 / 部分通过 / 建议修订 / 严重问题
- **主要发现**：列出 3–5 个最关键的问题或亮点
- **对 Story System 的影响**：
  - 是否需要未来在章节提交的 `health_flags` 中体现？
  - 是否需要在接下来的章节中用事件或设定补丁方式加以刻画？
- **下一步建议**：
  - 回到 `wuhang-ideation` / `wuhang-outline` / `wuhang-write` 的哪个步骤；
  - 是否建议在下一次运行 `project_doctor.py` 时重点关注哪些检查项。

## ⏭️ 下一步

- **质检通过**：
  - 若是设定质检 -> 加载 **wuhang-outline**（`wuhang-outline.md`）。
  - 若是大纲质检 -> 加载 **wuhang-write**（`wuhang-write.md`）。
  - 若是正文质检 -> 继续下一章 **wuhang-write**（`wuhang-write.md`）循环，并在后续提交中体现健康标记。

- **质检未通过**：
  - 根据建议修改对应文件（设定/大纲/正文），必要时执行大纲动态修订或重写章节；
  - 视情况在章节提交或 memory 中记录本次审查发现的问题与后续修正策略。
