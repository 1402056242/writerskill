## 📊 步骤 1：故事梗概提取

| 项目 | 内容 |
| :--- | :--- |
| **输入** | 样板书 `.txt` 路径（步骤 0 已确认） |
| **输出** | `1-边界/1.1_全书故事梗概.md`（全书或开篇梗概） |

**⚠️ 执行前须先与用户确认**：本步有两种方式，**二选一**。在执行前，必须先明确询问用户样板书的长度，并给出以下建议：

- **方式 A：Agent 循环提取（适合短篇/开局，无需额外 API）**
  - **适用**：样板书低于 30–50 章，或只拆解开篇。
  - **说明**：由 Agent 直接在对话中分批读取正文并总结。

- **方式 B：脚本批量生成（推荐长篇，需 API）**
  - **适用**：样板书高于 30–50 章。
  - **说明**：长篇用 Agent 容易导致上下文爆炸，建议用脚本跑全书梗概（需配置 API，有 Token 成本）。

待用户选择后，再按对应方式执行。

### 方式 A：Agent 循环提取（无需外部 API，仅消耗对话额度）

- **执行逻辑（AI 必须严格遵守）**：
  1. **分批提取与实时写入**：AI 调用提取脚本，每次读取 **5 章** 正文：
     ```bash
     python3 .soloent/skills/wuhang-long-all/scripts/extract_chapters.py 1-边界/[书名].txt 1 5
     ```
     阅读终端输出后，AI 必须**分章**提取梗概。每个章节独立输出一段梗概（约 50–100 字，重点概括核心事件、关键人物行动、剧情转折、伏笔），格式须为：
     ```markdown
     ## [完整章节标题]
     [这里是梗概内容...]
     ```
     将这 5 章的梗概**追加写入** `1-边界/1.1_全书故事梗概.md`（防止遗忘）。
  2. **循环执行**：修改命令参数（如 `6 10`、`11 15`、`16 20`、`21 25`、`26 30`），重复上述提取和追加写入动作，直到跑完前 30 章（共循环 6 次）。
  3. **全局自检与查漏补缺**：30 章全部写入后，AI 必须通读一遍 `1.1_全书故事梗概.md`，检查各章剧情逻辑是否连贯、是否有重要人物或道具交代不明（如“突然出现的法宝”）。
  4. **单章精准提取**：如果发现逻辑断层或遗漏，AI 需主动调用脚本提取对应单章（如 `python3 ...extract_chapters.py ... 15 15`）进行查证，并补充到梗概中。
  5. **强制审查点**：查漏补缺完成后，AI 必须**主动暂停**，向用户输出：
     > 「前 30 章的梗概已提取并自检完毕，请查看 `1.1_全书故事梗概.md`。看看是否还有遗漏？如果有，请告诉我进一步补充；如果没有，请回复『完成』结束本步骤，或回复『继续』提取后续章节。」
  6. 根据用户指令，补充内容或结束本步骤。
- **输出**：写入 `1-边界/1.1_全书故事梗概.md`，可在文件中注明「开篇梗概，非全书」。后续若用户提供 API，可再跑脚本覆盖或追加为全书梗概。

### 方式 B：脚本批量生成全书梗概（需 API）

按以下顺序在**对话中**完成，不要跳过步骤。

**1. 提醒用户注意**

- 样板书很长时 Token 消耗大（武行之前拆过自己的 200 章小说，花费 3–6 美金），可只处理部分章节或选用低成本 API。
- 使用 `.env` 配置 API 最安全：密钥仅存于本地文件、不会出现在命令行或聊天记录中，也不会被提交到仓库。

**2. 若用户尚未配置 API：先让脚本创建 `.env` 模板**

- 在**项目根目录**执行下面的命令（可由 AI 通过 Shell 代为执行一次）：
  ```bash
  python3 .soloent/skills/wuhang-long-all/scripts/summarize.py 1-边界/[书名].txt
  ```
- 脚本检测到缺少 API Key 时，会**自动在项目根创建 `.env` 文件**（模板）。
- 执行后，**在对话中明确提示用户**：「已在项目根创建 `.env` 文件，请打开该文件（路径：`项目根/.env`）查看并准备填写。」

**3. `.env` 命名与降级约定**

- 推荐 `.env` 中使用以下字段（与 boundary / 其他脚本统一）：
  - `NOVEL_API_BASE`
  - `NOVEL_API_MODEL`
  - `NOVEL_API_KEY`
- 如果将来引入 RAG，也可在同一 `.env` 中加入：
  - `EMBED_BASE_URL` / `EMBED_MODEL` / `EMBED_API_KEY`
- **降级策略**：
  - 若检测到 `NOVEL_API_KEY` 为空或未配置，`summarize.py` 必须**优雅退出**，给出清晰提示：「未检测到 API Key，将降级为 Agent 循环提取模式（方式 A）。」
  - 禁止在缺少 API Key 时硬报错或中途写入残缺文件。

**4. 提示用户打开并填写 `.env`**

- 在对话中说明：请打开项目根目录下的 **`.env`** 文件，在 `NOVEL_API_KEY=` 后面填入你的 API 密钥；如需更换服务商，可改 `NOVEL_API_BASE` 和 `NOVEL_API_MODEL`。**仅限本地使用，请勿将 `.env` 提交到仓库。**
- 请用户填写并保存后，在对话中回复「已填写」或「没问题」。

**5. 确认后执行脚本**

- 确认样板书已放入 `1-边界/` 下，且用户已确认 API 填写无误。
- **由 AI 在项目根代为执行脚本命令**（或给出命令供用户执行）：
  ```bash
  python3 .soloent/skills/wuhang-long-all/scripts/summarize.py 1-边界/[书名].txt
  ```
- 脚本会从项目根或 `.novelkit/.api.env` 读取 `.env` 配置，请勿在命令行中填写或粘贴 API Key。

**结果**：生成 `1-边界/1.1_全书故事梗概.md`。

## 🔍 步骤 5：微观节奏与剧情单元拆解

| 项目 | 内容 |
| :--- | :--- |
| **输入** | 样板书指定章节正文（前三章 + 一个完整高潮单元约 3–10 章）、`1.1_全书故事梗概.md` |
| **输出** | `1-边界/1.5_微观节奏拆解.md` |

### 子步骤 5.1：选材与样本提取

- **AI 动作**：
  1. **寻找剧情单元样本**：浏览 `1.1_全书故事梗概.md`，寻找字数占比适中（约 3–6 章）、起承转合明显的完整事件。推荐优先选择“新手村第一次大危机/大收获”（通常在第 10–20 章左右）。
  2. **提取对应正文**：根据锁定的章节号，调用 `extract_chapters.py` 脚本提取该剧情单元的正文。
  3. **提取黄金三章**：调用脚本提取样板书第 1、2、3 章正文，用于后续拉片。

### 子步骤 5.2：单剧情单元拆解（标准网文四步法）

- **目标**：分析作者如何用几章的篇幅讲好一个完整的小故事（小高潮）。
- **分析模型**（针对 5.1 提取的剧情单元）：
  1. **起 (Setup)**：欲望与阻碍。主角想要什么？谁在阻拦？（拉仇恨阶段）
  2. **承 (Build-up)**：试探与压抑。主角尝试解决但失败，或者被进一步羞辱/打压（情绪积蓄阶段）
  3. **转 (Climax)**：爆发与反转。金手指介入，或者智慧破局，形成最大反差（释放阶段）
  4. **合 (Resolution)**：收获与余波。战利品清点，地位提升，引出下一个麻烦（落袋为安阶段）

### 子步骤 5.3：单章精密拉片（以「黄金三章」为例）

- **目标**：显微镜级分析。针对前三章，看作者是如何留住读者的。
- **分析维度**（针对 5.1 提取的第 1–3 章，逐章分析）：
  1. **切入点**：第一段写了什么？环境/对话/危机/金手指？
  2. **信息密度控制**：什么时候抛出世界观？什么时候抛出设定？
  3. **场景结构分析**：
     - **场景组合**：单章通常包含几个场景？（例：1 大对话 + 1 小动作 + 1 内心独白）
     - **场景写法**：每个场景侧重什么？（心理博弈/环境渲染/动作描写/信息交代）
     - **镜头语言**：作者如何切换场景？（空镜头转场/时间跳跃/“镜头”拉远拉近）
     - **场景缝合**：空间转换怎么处理？（硬切 vs 软切）
     - **视角游移**：为展示主角的强/惨，视角是否切换到了旁人身上？
  4. **断章钩子**：这一章结尾最后一句话是什么作用？

### 子步骤 5.4：输出与确认

- **AI 动作**：将 5.2 和 5.3 的分析结果整合写入 `1-边界/1.5_微观节奏拆解.md`。
- **【强制暂停】** 输出以下提示后，**禁止继续执行任何后续步骤，等待用户明确回复**：
